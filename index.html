What is jQuery?

  • a library written in JavaScript…
  • …to help browser-based JavaScript development

jQuery is not…

  • a language
  • essential

It's not:

{NEXT} A language in it's own right.

jQuery syntax is JavaScript syntax.

It has a slightly different feel in some respects but it's still native JavaScript below the surface.

{NEXT} Neither is jQuery an essential part of modern web development.

Although you'd be forgiven for thinking so.

Like any library or framework, jQuery can help you write code

but the same functionality can be implemented without jQuery should you need to do so.

Why we use jQuery?

Ten years ago…

  • client-side heavy applications were new
  • but development was tough
  • browsers were buggy and inconsistent
  • for many, JavaScript was new and strange

Let's look at the history of jQuery.

In 2004, developers became interested in client-side browser applications.

Ajax was increasingly popular. And modern web apps such as GMail and Google Maps had started to appear.

{NEXT} But behind the scenes, browsers were a hostile environment.

Microsoft had abandoned Internet Explorer but it was still the most-used browser.

There were few tools and the best you could hope for was cryptic messages like this.

{NEXT} we had a couple of competing browsers such as Firefox and Opera

but they adopted newer W3C standards and weren't necessarily compatible with IE.

{NEXT} also, for many developers coming from backend or desktop languages, JavaScript seemed a little weird and difficult to comprehend.

JavaScript libraries

In 2004 developers had a handful of desktop browsers to support.

So they created helper functions which smoothed over the inconsistencies.

And these functions were wrapped into into libraries and released into the open source community.

We saw the launch of projects such as Prototype and MooTools which made development easier.

But then the iPhone, Android and other mobiles started to arrive.

We suddenly had numerous different devices, browsers, APIs and bugs to content with.

Using a JavaScript library became essential.

The rise of jQuery

// a single object for everything
$;
// CSS selector syntax
$("#main-section .name");
// concise method chaining
$("#main-section .name").text("hello").addClass("new").slideDown();

Now jQuery was launched in 2006 and arrived a little late to the party.

It's developer - John Resig - learned from the competing libraries which had started to become large and cumbersome.

He devised a number of elegant features.

First everything you do in jQuery is handled via single object - normally represented by the dollar sign.

It's simple and ties functionality to the HTML elements you want to interact with.

{NEXT} next, you could select elements from the DOM using CSS selectors.

Now this seems obvious now but it was revolutionary at the time.

{NEXT} jQuery offered method chaining.

Most methods return the jQuery object itself. This means it can be passed into another method to act on the same set of elements.

So here we have a selection of elements where we're setting the text to "Hello", adding a class, and running a slideDown animation.

It's concise. It's easy to read. Even developers with little experience could understand it.

This made jQuery phenomenally successful.

The problems jQuery solved for us don't necessarily exist today

So let's look at the bad parts.

{NEXT} you no longer need jQuery to locate nodes by CSS selectors.

Native CSS Selectors

// returns an array-like NodeList
var n = document.querySelectorAll("#myid .myclass");

// returns the first matching element
var f = document.querySelector("#myid .myclass");

All modern browsers have implemented methods named querySelector and querySelectorAll which find nodes using CSS selectors just like jQuery.

{QUESTION} Does anyone know which version of IE first implemented this?

It's IE8 - although you can only use simpler CSS2.1 selectors

Below the surface, jQuery uses when querySelectorAll when it can.

In effect jQuery is a simple wrapper around it.

So going back to our bad parts...

method chaining looks cool. Until you realise every method loops through every selected node. If you have three methods, that's three loops on the same elements. A single loop should be more effective.

{NEXT} next: jQuery doesn't support every browser.

It's great until you encounter a problem with a specific device. It doesn't happen often but when it does, it can be difficult to fix.

{NEXT} Next problem. And I think you can see where I'm going with this...

As you saw with chaining, jQuery allows you to implement functionality which works but is spectacularly inefficient.

Unfortunately, the code looks concise because jQuery abstracts the implementation.

{NEXT} what can be wrong with great documentation?

Visit StackOverflow and ask a question about JavaScript or any other library. 99 times out of 100 you'll get the answer "seriously dude, just use jQuery".

jQuery documentation is ubiquitous and that can make it difficult to find information when you're using something else.

{NEXT} the jQuery core ONLY covers the core APIs. But these are consistent in modern browsers and there are dozens of HTML5 APIs it won't help you with. In fact, you may struggle to use a new API because your jQuery crutch has been kicked away.

{NEXT} even plugins will only build on the jQuery core. Now you'll find plugins which don't use the core but then -- why are they using jQuery? A self-contained module would be compatible with any library.

jQuery Misconceptions

  • jQuery is perfect
  • write less, do more
  • jQuery is only 30Kb
  • learning JavaScript isn't necessary


Are we using jQuery out of habit rather than necessity?

jQuery's success has also led to several misconceptions within the community.

{NEXT} jQuery has bugs and issues like any other project. It is robust, but don't expect it to be fault-free.

{NEXT} this is jQuery's tag line but I don't think it's always true today.

{NEXT} minified and gzipped jQuery is only 32K. And that's nothing when compared with assets like images.

In reality, the uncompressed source is nearer 300K and that has a processing and storage overhead.

We rarely use use more than a fraction of jQuery's functionality yet the whole library is referenced in every object you create.

{NEXT} finally, the classic: "learning JavaScript isn't necessary because jQuery exists"

Developers may think they understand JavaScript when they know a little jQuery - but they'd struggle without it.

And don't forget that jQuery may not be around forever. Ask any YUI developer...

{NEXT} So are we using jQuery out of habit rather than necessity? That's something I've been guilty of.

Raw JavaScript

  • has become considerably easier
  • has become cool (again)
  • it's considerably faster

So let's look at the benefits of raw JavaScript.

{NEXT} the core browser APIs are consistent now that oldIEs have died.

{NEXT} everyone's jumping on the JavaScript bandwagon. It can be used outside the browser - such as on the server in Node.js, in OSes and even embedded systems.

{NEXT} and it's fast. It's far faster than using jQuery which, as we've seen, is often a wrapper around native methods.

This performance is increasingly important.

{QUESTION} How many you are working on sites with a responsive web design?

We spend many development hours making a site design responsive but put relatively little effort into efficiency or performance.

The design may be responsive but the user experience isn't.

So to prove this, let's look at the performance gains you can make by removing your jQuery shackles and running naked and free in the JavaScript world!

Class manipulation

TEST: add two classes, toggle one & remove other on all LI's

// 1: jQuery single
$li.addClass("test1"); $li.addClass("test2");
$li.toggleClass("test2"); $li.removeClass("test1");
// 2: jQuery chain
$li.addClass("test1").addClass("test2").toggleClass("test2").removeClass("test1");
// 3: classList API, multiple loops
for (i = 0; i < li.length; i++) { li[i].classList.add("test1"); }
for (i = 0; i < li.length; i++) { li[i].classList.add("test2"); }
for (i = 0; i < li.length; i++) { li[i].classList.toggle("test2"); }
for (i = 0; i < li.length; i++) { li[i].classList.remove("test1"); }
// 4: classList API, single loop
for (i = 0; i < li.length; i++) {
  li[i].classList.add("test1");
  li[i].classList.add("test2");
  li[i].classList.toggle("test2");
  li[i].classList.remove("test1");
}

In the next test we're going to modify all LIST elements on the page.

We'll add two class names, toggle one and removing the other.

Class manipulation is something you do regularly when building client-side applications.

In all these tests the collection of HTML elements has been cached so there's no additional overhead.

for the first test, we'll run jQuery methods as separate commands.

{NEXT} in the second test, we'll improve this by chaining the methods together.

{NEXT} in the next test, we're going to use the new classList API which is available in IE10 and above. It offers similar add, remove and toggle methods.

We don't have the benefit of jQuery doing the loop for us, so we'll do four loops - one for each change of class.

{NEXT} and in the final test, we'll still use the classList API but do everything in a single loop which you'd expect to be more efficient.

Performance gains

Similar gains can be made for:

  • arrays
  • loops
  • event handling
  • Ajax

These gains will be repeated elsewhere:

arrays have new methods such as map and every which can be more efficient than the loops we used earlier

{NEXT} jQuery's .each() method can be replaced using JavaScript's faster forEach statement

{NEXT} event handling is consistent so you rarely need the overhead of a jQuery wrapper

{NEXT} and similarly, jQuery's Ajax handling is quite complex.

it caters for multiple situations and you're unlikely to require them all.

In IE10 and above you can use XMLHttpRequest2.

which has advanced methods for things like progress bars, files and binary data which jQuery won't help you with.

In fact, there's only one exception I'd make...

Animations

Don't bother with jQuery or JavaScript

use CSS3

that's animations

jQuery offers simple slides, fades and scrolling.

They work in all browsers, but really - just forget jQuery and JavaScript.

{NEXT} Use CSS3 animations and transformations.

it's quicker to write, faster, more sophisticated and uses fewer resources than whatever you come up with.

Now CSS3 animations are simplistic but they're good enough for basic website effects.

You don't get the frame control you need for action games but there are better options than jQuery

such as Velocity or GSAP.

Reasons to retain jQuery

  • legacy applications
  • where IE6/7 support is required
  • projects using lots of jQuery functionality
  • quick and dirty projects or prototypes
  • large teams with varying client-side skills

OK, so while performance could be an issue, I'd never say you should never use jQuery. It's still a great library.

{NEXT} there's little point ripping it out of legacy applications unless you have a serious performance issue

{NEXT} if you absolutely have to support IE6 and 7, well, I pity you. jQuery will help - just remember to stick with version 1.

You could argue it's good for IE8 support but it's more marginal

{NEXT} if you're using a high proportion of jQuery functionality. But does your project really use every DOM methods, every Ajax type and IE8 animations?

{NEXT} perhaps you could use it for throwaway prototypes where performance isn't so much of an issue.

You could use it on projects where a client wants something delivered quickly at low cost and doesn't care about quality ... but that's soul destroying work.

{NEXT} finally, large teams with varying skills is another possibility. But be careful not to choose jQuery because it's the easy option.

You might be better to offer training, ahem, hire a freelance client-side developer.

But overall, though, my main advice would be...

jQuery alternatives

  • Zepto.js - 9Kb
  • Minified.js - 4Kb to 8Kb
  • min.js - 600 bytes
  • or create a specific, modular build of jQuery

Admittedly, it takes time to become a JavaScript Jedi Master. Even then, you'll still want helper functions to manage things like Ajax.

If you don't want the hassle of writing those yourself or you cannot imagine life without jQuery's syntax

there is a compromise in the form of micro-libraries.

these mimic jQuery's API in a smaller package with better performance

as you can see, min.js is only 600 bytes and is essentially a wrapper for querySelectorAll and events

and jQuery itself offers custom builds. You can omit parts you don't need although it won't lead to a significant performance gain.
